## Plan: 基于 Spring Boot 的 AI 智能股票预测与模拟交易系统实施方案

当前项目已经完成了后端基础骨架与少量账户/股票模块，但整体规划仍偏粗。为了避免后续出现“先开发页面、后返工数据库”的问题，现将项目调整为**先完成业务模型与数据库基线，再推进交易、预测、前端联调**的实施路线。

**Steps**

**1. 阶段一：项目基线整理与核心模型收敛（当前优先级最高）**
*   统一梳理业务边界，明确系统定位为：**AI 预测 + 行情展示 + 模拟交易 + 用户资产管理**。
*   明确后端主技术栈：Spring Boot 3、JDK 17、MyBatis-Plus、MySQL、Redis、JWT、Knife4j。
*   统一工程分层：`controller`、`service`、`mapper`、`entity`、`dto`、`vo`、`config`、`common`。
*   统一数据库命名规范：**所有业务表统一使用 `zidatrade_` 前缀**，字段命名使用下划线风格，时间字段统一为 `created_at`、`updated_at`。

**2. 阶段二：数据库结构重构与初始化脚本完善（当前核心交付）**
*   重构 `backend/src/main/resources/sql/init.sql`，保留已有核心表并进行结构升级。
*   调整后的核心主表包括：
    *   `zidatrade_user`：用户账户、资产、角色、风险等级、最近登录时间。
    *   `zidatrade_stock_info`：股票基础资料、市场、行业、上市日期、交易状态。
    *   `zidatrade_trade_order`：委托订单、方向、类型、成交数量、冻结金额、订单状态。
    *   `zidatrade_user_position`：用户持仓、可卖数量、冻结数量、成本价、市值、浮盈亏。
    *   `zidatrade_ai_predict_log`：模型预测结果、置信度、预测信号、验证状态。
*   新增支撑业务闭环的扩展表：
    *   `zidatrade_stock_daily_price`：股票日线行情，用于训练、回测与 K 线展示。
    *   `zidatrade_trade_deal`：订单成交明细，用于撮合结果留痕与成交查询。
    *   `zidatrade_user_asset_snapshot`：用户每日资产快照，用于收益曲线与风控展示。
*   为核心业务表补充索引、唯一约束与必要外键，保证查询性能与数据一致性。

**3. 阶段三：账户、行情与预测能力补齐**
*   完成注册、登录、个人信息、资产信息接口的稳定化。
*   增加股票列表、股票详情、历史行情查询接口。
*   预留 AI 预测结果查询接口，支撑“单只股票预测”和“预测历史回溯”场景。
*   通过定时任务或外部数据服务补充 `zidatrade_stock_daily_price` 数据。

**4. 阶段四：模拟交易核心链路建设**
*   实现买入、卖出、撤单、订单查询、成交记录查询。
*   以订单表、成交表、持仓表、资产快照表形成完整交易闭环。
*   基于 Redis 或数据库事务处理资金冻结、持仓冻结、部分成交与 T+1 可卖逻辑。

**5. 阶段五：AI 预测系统联动与前端可视化**
*   对接 Python/FastAPI 预测服务，按股票和日期写入 `zidatrade_ai_predict_log`。
*   前端继续采用 Vue3 + Vite + 暗黑专业交易风格。
*   在个股详情页叠加 K 线、预测线、成交量、信号标签。
*   在用户中心展示总资产、持仓分布、收益曲线、最近交易记录。

**6. 阶段六：联调、校验与上线准备**
*   校验数据库初始化脚本可重复执行。
*   校验后端现有实体映射、接口启动、SQL 日志输出正常。
*   为后续接口文档、测试数据、部署说明留出标准化空间。

**Relevant files**
*   根目录 `plan.md`：项目整体计划书与阶段目标。
*   `backend/src/main/resources/sql/init.sql`：数据库初始化脚本与表结构定义。
*   `backend/src/main/java/com/smarttrade/entity/`：MyBatis-Plus 实体映射定义。
*   `backend/src/main/resources/application.yml`：数据库连接与 MyBatis-Plus 配置。

**Verification**
1. 执行 `init.sql` 后，MySQL 中所有业务表均以 `zidatrade_` 开头，且主外键、唯一索引创建成功。
2. Spring Boot 项目启动后，用户注册、登录、股票列表接口可正常访问。
3. `User`、`StockInfo` 等现有实体能正确映射新表名，不因旧表名失效导致运行报错。
4. 新增表能够支撑后续行情采集、AI 预测写入、交易成交留痕和资产曲线展示。

**Decisions**
- 项目优先完成数据库与领域模型收敛，再继续扩展业务接口和前端页面，降低返工成本。
- 所有业务表统一使用 `zidatrade_` 前缀，便于后续部署、迁移和多项目隔离。
- 当前系统以模拟交易为主，因此暂不引入真实券商接口、印花税与复杂清算规则。
- 数据模型必须预留 AI 预测、历史行情、成交记录、资产快照四类核心数据能力，避免后续再次拆表。
